# Deploying Elements (Render free tier + MongoDB Atlas free tier)

Two ways to run Elements:

| Mode | What you get | Cost |
|---|---|---|
| **Static only** | Full app: table, journey, games, lab, badges. Progress in localStorage. No accounts. | Free anywhere (Render Static Site, GitHub Pages, Netlify…) |
| **With companion server** | Everything above **plus** optional Google sign-in, cross-device progress sync, and the institution admin dashboard. | Free (Render free web service + MongoDB Atlas M0) |

The app is **local-first**: even with the server, kids who never sign in
lose nothing — progress stays in localStorage. Signing in *adds* a synced
copy (merge = union of achievements; nothing is ever overwritten away).

---

## Option A — Static site (5 minutes)

1. Render Dashboard → **New → Static Site** → connect this repo.
2. Build command: *(leave empty)* · Publish directory: `.`
3. Done. (Auth and the dashboard are hidden automatically in this mode.)

## Option B — Full deployment (server + auth + dashboard)

### 1. MongoDB Atlas (free M0)

1. Create a free cluster at [mongodb.com/atlas](https://www.mongodb.com/atlas)
   (M0 tier — free forever, no card needed).
2. **Database Access** → add a database user (username + password).
3. **Network Access** → *Allow access from anywhere* (`0.0.0.0/0`) —
   Render's free tier has no static outbound IPs.
4. Copy the connection string (**Connect → Drivers**), e.g.
   `mongodb+srv://USER:PASS@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority`
   The server uses database `elements` (override with `MONGODB_DB`).

### 2. Google OAuth client

1. [Google Cloud Console](https://console.cloud.google.com) → create a
   project → **APIs & Services → Credentials → Create Credentials →
   OAuth client ID → Web application**.
2. First configure the consent screen (External; app name "Elements";
   scopes: none beyond the default OpenID/email/profile).
3. **Authorized JavaScript origins**: add your Render URL, e.g.
   `https://elements-xxxx.onrender.com` (and `http://localhost:8080` for
   local testing). No redirect URIs needed (Google Identity Services popup flow).
4. Copy the **Client ID** (ends in `.apps.googleusercontent.com`).

### 3. Deploy on Render (free web service)

The repo contains a `render.yaml` blueprint:

1. Render Dashboard → **New → Blueprint** → select this repo.
2. When prompted for environment variables:
   - `GOOGLE_CLIENT_ID` — from step 2
   - `ADMIN_EMAILS` — comma-separated admin (teacher) Google emails,
     e.g. `principal@school.org,teacher@school.org`
   - `MONGODB_URI` — from step 1
   - `SESSION_SECRET` — auto-generated by the blueprint
3. Deploy. Health check: `https://YOUR-APP.onrender.com/api/health`
   should return `{"ok":true,"store":"mongodb"}`.

Manual alternative (no blueprint): **New → Web Service**, runtime Node,
build command `cd server && npm install --omit=dev`, start command
`node server/index.mjs`, plus the env vars above.

### 4. Set up your institution

1. Open the app, sign in with an email listed in `ADMIN_EMAILS`.
2. A **Dashboard** button appears in the header → add students by their
   school Google email.
3. Students sign in on their devices; their progress appears on the
   dashboard automatically (they can also keep playing without signing
   in — then they simply don't appear in analytics).

## Free-tier facts to know

- **Cold starts**: Render free web services spin down after ~15 min idle;
  the next visit takes ~30–60 s to wake. Fine for classrooms; upgrade or
  use a cron ping if it bothers you.
- **Data**: all persistent data lives in MongoDB Atlas (M0 free tier has
  512 MB — a classroom's progress uses a few hundred KB). The Render
  filesystem is ephemeral and is *not* used for data when `MONGODB_URI`
  is set.
- **Local dev**: `node server/index.mjs` with no env vars serves the app
  on :8080 with a JSON-file store in `server/data/`. Add `DEV_FAKE_AUTH=1`
  to test sign-in flows without Google (never set this in production).

## Child-privacy notes (read before onboarding a school)

- Accounts are designed for **school-managed Google Workspace accounts**,
  added by an admin — there is no self-signup and no public account creation.
- The server stores: email, display name, and game progress. Nothing else.
  No analytics/tracking scripts of any kind.
- Anonymous localStorage mode remains the default for every visitor.
- This is engineering guidance, not legal advice — have your institution's
  compliance owner review COPPA/GDPR-K obligations before rollout.
